# Prioriser les tâches

## Matrice impact / effort
- Quick wins : fort impact, faible effort → d’abord
- Grands paris : fort impact, fort effort → planifier
- Remplissage : faible impact, faible effort → plus tard
- À éviter : faible impact, fort effort

## Ordre technique souvent optimal
1. Fondations (repo, env, CI minimale)
2. Modèle de données cœur
3. Auth si nécessaire
4. Parcours utilisateur principal (happy path)
5. Cas limites, admin, polish

## Règle mentor
Priorise ce qui réduit le risque et débloque les apprentissages — pas ce qui est “intéressant”.
